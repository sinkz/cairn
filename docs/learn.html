OKF reference

Referência OKF

What Cairn borrows from Open Knowledge Format.

O que o Cairn aproveita do Open Knowledge Format.

Google describes OKF v0.1 as an open, vendor-neutral, human- and agent-friendly way to represent knowledge as directories of Markdown files with YAML frontmatter. In OKF, each concept is one file; the file path is its identity; frontmatter carries structured fields; the Markdown body carries the prose; links can turn the bundle into a graph; `index.md` and `log.md` can support progressive navigation and history.

O Google descreve o OKF v0.1 como um formato aberto, neutro de fornecedor, amigável para humanos e agentes, que representa conhecimento como diretórios de arquivos Markdown com frontmatter YAML. No OKF, cada conceito é um arquivo; o caminho do arquivo é sua identidade; o frontmatter carrega campos estruturados; o corpo Markdown carrega o texto; links podem transformar o bundle em um grafo; `index.md` e `log.md` podem apoiar navegação progressiva e histórico.

Algorithm and writeback

Algoritmo e escrita

Search and retrieval use a staged local pipeline.

Busca e recuperação usam um pipeline local em etapas.

The default search path is strict BM25 over the local SQLite index. If a top-level `glossary.md` exists, approved aliases are tried only after the exact query returns no rows. RRF can be requested directly, and `retrieve --ranker auto` tries BM25 first before falling back to RRF when no context is returned.

O caminho padrão é BM25 estrito sobre o índice SQLite local. Se existir um `glossary.md` no topo do vault, aliases aprovados são testados só depois que a consulta exata não retorna linhas. RRF pode ser pedido diretamente, e `retrieve --ranker auto` tenta BM25 primeiro antes de cair para RRF quando nenhum contexto é retornado.

index

Frontmatter + body

Title, description, tags, aliases, systems, signals, body, and passages are indexed locally.

Título, descrição, tags, aliases, sistemas, sinais, corpo e passagens são indexados localmente.

rank

BM25 / RRF / auto

BM25 handles strict ranking. RRF recovers noisy queries. Auto tries BM25 first.

BM25 faz o ranking estrito. RRF recupera queries ruidosas. Auto tenta BM25 primeiro.

A recuperação encaixa conteúdo em um orçamento aproximado de tokens e redige valores com aparência de segredo.

Benchmark methodology

Metodologia de benchmark

The benchmarks measure retrieval and writeback before a model gets involved.

Os benchmarks medem recuperação e escrita antes de envolver um modelo.

Fixtures define vault notes, queries, expected relevance labels, golden result prefixes, retrieval mode, comparison mode, and writeback decisions. The goal is to catch regressions in search, token usage, and update-vs-create behavior deterministically.

Fixtures definem notas do vault, queries, rótulos esperados de relevância, prefixos golden, modo de recuperação, modo comparativo e decisões de escrita. O objetivo é capturar regressões em busca, uso de tokens e comportamento de atualizar vs criar de forma determinística.

Recall@3

Fração dos tópicos em que pelo menos uma nota esperada aparece nos três primeiros resultados.

MRR@3

Rewards the first relevant result appearing earlier in the list.

Recompensa quando o primeiro resultado relevante aparece mais cedo na lista.

nDCG@3

Checks whether highly relevant results are ordered above weaker matches.

Verifica se resultados muito relevantes ficam acima de matches mais fracos.

context_reduction

Measures how much less text passage retrieval returns compared with full documents.

Mede quanto texto a recuperação por passagens economiza contra documentos completos.

decision_accuracy

Checks whether a writeback case correctly creates, updates, no-ops, or blocks a conflict.

Verifica se um caso de escrita cria, atualiza, faz no-op ou bloqueia conflito corretamente.

duplicate_avoidance

Verifica se notas reutilizáveis existentes são reaproveitadas em vez de duplicadas.
